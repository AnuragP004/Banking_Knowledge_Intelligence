\section{Overview}
The Viniyog One project runs atop a strictly pruned, authoritative dataset architecture built to replicate a realistic internal Indian Banking Knowledge Base. 

\subsection{Dataset Artifact Details}
\begin{itemize}
    \item \textbf{Path:} \texttt{data/banking\_schemes.json}
    \item \textbf{Format:} Structured JSON Array
    \item \textbf{Total Entities:} 15 Complete Financial Products
\end{itemize}

\section{Structural Fields}
Every mapped entry conforms strictly to a unified schema required by the ingestion layer. This ensures volatile quantitative facts (interest rates, fixed deposit yields) are not lost against unstructured vector noise distributions.

\section{System Interaction}
The integration layer parses this fundamental definition via three sequential loops:
\begin{enumerate}
    \item \textbf{Idempotent Bootstrapping:} On application lifecycle startup, \texttt{ingestion\_service.py} evaluates the hash signature of \texttt{banking\_schemes.json}. If variations or updates exist, the pipeline natively triggers a vector reconstruction operation.
    \item \textbf{Text Generation Optimization:} Because standard dense retrievers (like MTEB-leader \texttt{BGE}) process prose much cleaner than raw JSON properties, our ingest logic parses the structural arrays and re-templates them into active Markdown narrative formatting (e.g., transforming \texttt{"eligibility": ["Over 18"]} into \texttt{\#\# Eligibility \textbackslash n - Over 18}).
    \item \textbf{Database Mirroring (KB Fallback):} Simultaneously, the unadulterated JSON array dynamically imports directly into the isolated SQLite \texttt{data/knowledge\_base.db} table. This cleanly powers the secure \texttt{/kb/token} authorization pipeline via rigid \texttt{FTS5} (Full-Text Search) keyword queries.
\end{enumerate}

1. PROJECT OVERVIEW
================================================================================

WHAT: Enterprise-Grade RAG System for a banking domain.
WHO: Take-home assessment for Viniyog One.
CORE PRINCIPLE: The LLM is NOT the source of truth. The Knowledge Base (KB) is.

The system must:
  - Answer banking queries using RAG (retrieval-augmented generation)
  - Detect hallucinations via an LLM-as-a-judge mechanism
  - Fall back to a secure Knowledge Base when hallucination is detected
  - Expose evaluation-ready APIs with RBAC and observability
  - Justify every architectural decision in documentation

Full BRD is in the original prompt / README (to be expanded).

4. KEY ARCHITECTURAL DECISIONS — RATIONALE
================================================================================

DECISION 1: Header-based RBAC simulation (not real OAuth2)
  WHY: Assessment needs to be testable without an identity provider.
  HOW: X-User-Role header simulates role. Dependency injection means
       replacing get_current_user() with real OAuth2 requires ZERO
       endpoint code changes.
  TRADEOFF: Not secure for production. Documented as such.

DECISION 2: Separate KB token from user auth
  WHY: Even an authenticated admin should not directly access KB without
       a fresh, purpose-scoped token. Follows Principle of Least Privilege.
  HOW: /kb/fetch requires BOTH system role AND a valid JWT from /kb/token.
  TRADEOFF: Extra hop, but enforces the "KB is the source of truth" principle.

DECISION 3: HS256 for JWT (not RS256)
  WHY: Single-service architecture — no public key distribution needed.
       Lower computational overhead for high-frequency token generation.
  TRADEOFF: In a microservices setup, RS256 with JWKS would be preferred.

DECISION 4: Short-lived KB tokens (5 minutes)
  WHY: Minimizes blast radius of leaked tokens. In banking, time-limited
       access is a regulatory expectation.
  TRADEOFF: System must re-request tokens frequently. Acceptable for
       the fallback pattern (only triggered on hallucination).

DECISION 5: Application Factory pattern (create_app)
  WHY: Enables creating multiple app instances for testing, separates
       config from instantiation, follows FastAPI best practices.

DECISION 6: Loguru over stdlib logging
  WHY: Zero-config structured output, built-in rotation/retention,
       exception formatting. Critical for banking observability requirements.

DECISION 7: pydantic-settings over dotenv
  WHY: Type validation on config (fail-fast), consistent API with
       Pydantic models already used in schemas, env-var override built-in.

DECISION 8: ChromaDB over FAISS as primary
  WHY: ChromaDB provides metadata filtering, persistence, and a friendlier
       API. FAISS is available as fallback for pure speed benchmarks.

DECISION 9: FTS5 with OR logic (not AND) for KB search
  WHY: KB is the FALLBACK — when we hit it, vector search already failed.
       We need HIGH RECALL. OR logic ensures we don't miss results.
       BM25 ranking naturally scores documents matching more terms higher,
       so precision is maintained via ordering, not filtering.
  TRADEOFF: May return loosely related schemes, but top results are
       always most relevant thanks to BM25.

DECISION 10: Deterministic text search for KB (not vector search)
  WHY: KB is the fallback when vector search fails. Using the same
       vector approach would risk the same failure mode. Deterministic
       FTS5 + LIKE fallback guarantees results for any query.
  TRADEOFF: No semantic understanding, but compensated by comprehensive
       scheme descriptions in the JSON data.

KEY DECISIONS:
─────────────────────────────────────────────────────────────────────────────────

  DECISION 14: Dual-layer hallucination detection
    WHY: Heuristics are fast (<1ms) and catch obvious issues.
         LLM judge is thorough but slow and costs tokens.
         Combined scoring gives reliability at reasonable cost.

  DECISION 15: Banking-specific system prompt guardrails
    WHY: The prompt explicitly forbids fabricating rates, schemes,
         or eligibility. This is a legal requirement in banking
         advisory — incorrect information can cause customer harm.

  DECISION 16: Temperature 0.1 for generation
    WHY: Banking requires factual accuracy, not creativity.
         Low temperature reduces randomness in outputs.
         Higher temperatures (0.7+) would risk fabricated details.

  DECISION 17: Deterministic fallback when no LLM
    WHY: System must be functional even without OpenAI.
         Returns raw context from vector search — zero hallucination
         possible. This is the safest mode for banking data.

     File: app/services/vector_store_service.py
     What: Two-stage retrieval — initial Top-K=10 via bi-encoder (BGE),
           then cross-encoder re-ranking (ms-marco-MiniLM-L-6-v2) to Top-3.
     Why: Bi-encoders embed query and document SEPARATELY (fast but approximate).
          Cross-encoders process them TOGETHER through attention layers,
          computing deep token-level interactions for much higher precision.
          Adds ~50ms per query — acceptable for banking.
     Model: cross-encoder/ms-marco-MiniLM-L-6-v2 (22M params, fast)

  2. HALLUCINATION THRESHOLD RAISED (0.7 → 0.85)
     File: app/core/config.py
     What: Stricter groundedness threshold for banking domain.
     Why: 0.7 was too permissive for financial data. 0.85 catches subtle
          fabrications. The KB fallback is reliable, so more false positives
          are acceptable. In banking, false negatives are catastrophic.

  3. STRUCTURED XML CONTEXT ASSEMBLY
     File: app/services/rag_pipeline.py
     What: Context injected with <source id="1" scheme="..."> XML tags
           instead of plain text separators (---).
     Why: LLMs parse XML tags more reliably. Explicit source IDs enable
          better attribution. The model can reference specific chunks.

  4. WIDER INITIAL RETRIEVAL (5 → 10)
     File: app/core/config.py
     What: RETRIEVAL_TOP_K changed from 5 to 10.
     Why: Combined with cross-encoder re-ranking, a wider initial pool
          catches adjacent policy clauses. Without re-ranking this would
          hurt (context dilution), but with re-ranking it's strictly better.

  5. ATOMIC CLAIM DECOMPOSITION IN JUDGE
     File: app/services/hallucination_detector.py
     What: Judge prompt now decomposes responses into individual atomic
           claims (each fact, number, name, date) and verifies each.
     Why: Catches partial hallucinations (e.g., correct scheme name but
          fabricated rate). More granular than overall scoring. Outputs
          CLAIMS_SUPPORTED, CLAIMS_UNSUPPORTED, and UNSUPPORTED_DETAILS.

  6. REDUCED SYSTEM TOKEN EXPIRY (5 min → 60 seconds)
     File: app/core/config.py
     What: KB_TOKEN_EXPIRY_MINUTES changed from 5 to 1.
     Why: KB fallback completes in <5 seconds. 60s is generous. Tighter
          security reduces blast radius of leaked tokens.
